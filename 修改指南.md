# check_price 修改指南

更新日期: 2026-03-17

## 目標

這份文件整理目前專案較值得優先處理的修改方向，重點不是重寫整個系統，而是先補強會影響實際使用、維護成本與後續擴充的部分。

## 目前專案定位

這個專案已經不只是單次查價腳本，而是一套包含以下能力的交易監控工具：

- 即時分析
- 本地 API 與簡易網頁
- Telegram 警報
- 狀態持久化
- 歷史回填與績效追蹤
- Markdown / JSON 報表輸出

因此修改策略應以「提高穩定性與可維護性」為先，而不是優先增加新功能。

## 優先修改項目

### 1. 修正警報送出失敗時的狀態一致性

影響檔案:

- `market_alert_daemon.py`

問題:

- 目前流程會先更新 `symbol_state`，再送 Telegram。
- 如果送訊失敗，symbol 的狀態可能已經從 `idle` 進到 `confirmed`、`retest` 或 `reconfirmed`。
- 但 alert 紀錄沒有成功寫入，使用者也沒有真的收到通知。
- 這會造成「系統以為已處理，但實際沒有通知」的漏警報風險。
- 若採用「同輪任一事件失敗就整個 symbol 回滾」的作法，還會引入另一個問題：前面已經成功送出的事件，下一輪可能被重送。

建議修改方向:

- 將「狀態更新」與「通知成功後的記錄」重新排序。
- 不建議用籠統的「失敗就回滾 state」描述需求，因為這會讓已成功送出的事件變成下一輪重送候選。
- 較安全的原則是：
- 先計算 `events` 與 `next_state`
- 將本輪需要發送的事件全部送完
- 只有在本輪事件都成功處理後，才正式提交 `next_state`

建議實作方式:

- 每個 symbol 先算出 `events` 與 `next_state`。
- 逐一發送事件。
- 每個成功送出的事件，照常寫入 alert record。
- 若本輪還有未送出的事件失敗，保留舊的 symbol state，不提交 `next_state`。
- 只有在本輪事件全部成功處理後，才更新 `symbol_states`。

驗證重點:

- 模擬 Telegram 腳本不存在或 HTTP token 缺失。
- 確認送訊失敗時，不會把 state 推進到下一階段。
- 修正後重新跑 `--once`，確認成功送訊時 state 與 alert record 都正常更新。

### 2. 修正 legacy alert 遷移時的方向判定

影響檔案:

- `alert_performance_tracker.py`

問題:

- `migrate_legacy_alerts()` 會依賴 `event_direction()` 決定事件方向。
- 目前 `breakout_touch_up` / `breakout_touch_down` 沒有被映射到正確方向。
- 舊資料遷移後，這兩種事件可能被算成 `neutral`，導致績效統計失真。

建議修改方向:

- 將 `breakout_touch_up` 納入 `up`。
- 將 `breakout_touch_down` 納入 `down`。
- 最好讓 daemon 與 tracker 共用同一份事件方向映射，而不是各自維護。

可進一步改善:

- 抽出共用常數，例如 `EVENT_DIRECTION_MAP`。
- 避免未來新增事件類型時，兩邊定義不同步。

驗證重點:

- 建立一筆舊版 `alerts` 測試資料。
- 跑一次 `migrate_legacy_alerts()`。
- 確認對應事件在 `alert_events.direction` 內是正確方向。

### 3. 修正 API 頁面的 HTML 注入風險

影響檔案:

- `advisor_api_server.py`

問題:

- 頁面使用 `innerHTML` 直接插入外部資料。
- 資料來源包含 RSS 標題、分析文字、模型輸出。
- 這些內容理論上可能帶有 HTML 或 script 片段。

建議修改方向:

- 不要把未轉義字串直接拼進 HTML。
- 改用以下其中一種方式：
- 建立 DOM 節點後用 `textContent` 填值。
- 若仍要保留字串模板，至少加上 HTML escape 函式處理所有可變字串。

建議優先方案:

- 直接改成以原生 DOM API render。
- 因為欄位數不多，手工建立節點的成本可接受，安全性也更直觀。

驗證重點:

- 模擬某個欄位包含 `<script>alert(1)</script>`。
- 頁面應該只顯示文字，不執行 script。

### 4. 補強長時間執行腳本的錯誤隔離

影響檔案:

- `hourly_wait_reporter.py`
- `market_alert_daemon.py`
- 視情況也可檢查 `alert_performance_tracker.py`

問題:

- `hourly_wait_reporter.py` 主迴圈缺少循環級別的例外保護。
- `market_alert_daemon.py` 雖然有 symbol-level 保護，但整個 cycle 仍缺少 loop-level 保護。
- 任一次設定讀取失敗、Telegram 失敗、行情來源異常或其他未預期錯誤，都可能讓長時間執行程序直接退出。

建議修改方向:

- 在 `while True` 內包 cycle-level `try/except`。
- 發生錯誤時記錄 log，等待下一輪再繼續。
- 若錯誤是設定檔缺失這種不可恢復型錯誤，再考慮直接退出。
- `market_alert_daemon.py` 仍可保留 symbol-level 保護，但外層要再補一層，避免整輪直接中止。

驗證重點:

- 模擬 Telegram 失敗。
- 模擬設定檔暫時讀取失敗。
- 模擬某個資料來源 timeout。
- 確認程式會印出錯誤並在下一輪繼續跑。

## 第二優先修改項目

### 5. 補 `.gitignore`

目前專案缺少 `.gitignore`，但根目錄已有許多執行產物：

- `*.db`
- `*.log`
- `__pycache__/`
- `reports/`
- 本機環境檔

建議忽略項目:

- `__pycache__/`
- `*.pyc`
- `*.db`
- `*.log`
- `reports/`
- `.pi_telegram_env`
- `.pi_telegram_env_ascii`

若某些報表要保留版本歷史，可改為只忽略自動產生的檔名模式。

### 6. 建立依賴定義檔

目前沒有：

- `requirements.txt`
- `pyproject.toml`

至少應補一個最小版本，讓環境可重建。

若走簡單路線:

- 建立 `requirements.txt`

建議至少包含:

- `requests`

若後續想更完整管理專案:

- 建立 `pyproject.toml`
- 補上專案名稱、Python 版本與依賴

### 7. 補自動化測試

目前最值得先補的不是 UI 測試，而是核心規則測試。

建議優先測試範圍:

- `build_events()` 狀態轉移
- `event_direction()` 映射
- `extract_risk_levels()`
- `migrate_legacy_alerts()`
- `summarize_news()` 的基本輸出
- `build_short_term_signal()` 的多種分支

建議測試情境:

- long breakout 進入 `touched -> confirmed -> retest -> reconfirmed`
- short breakdown 對應流程
- false breakout 被過濾
- cooldown 生效
- Telegram 發送失敗時 state 不前進

補測試的目的，是讓後續調整警報 state、tracker 與報表時有最低限度的回歸保護。

### 8. 把共用邏輯抽成模組

目前多個腳本彼此直接 import，短期可行，但長期會遇到這些問題：

- 共用常數散落
- 事件類型與方向對照容易不一致
- schema 定義重複
- 測試時難以只測單一邏輯

建議拆分方向:

- `core/market_data.py`
- `core/analysis.py`
- `core/alerts.py`
- `core/reporting.py`
- `core/db.py`

但這件事不建議排在測試之前。

原因:

- 目前 repo 還沒有測試骨架。
- 若先大幅搬動到 `core/*`，很難確認沒有打壞 alert state、tracker 與 report generation。

短期不需要大規模搬移，可先抽：

- 事件方向映射
- DB schema 初始化
- Telegram 發送包裝

## 第三優先修改項目

### 9. 統一設定與常數

目前很多值直接散在程式裡，例如：

- volume ratio 門檻
- proximity 百分比
- timeout
- ATR 倍數
- second breakout / failure / scale out 設計

建議做法:

- 將策略常數集中成設定物件或獨立模組。
- 區分「使用者設定」與「策略內部常數」。

這樣的好處:

- 調參更集中
- 回測與實盤比較時更容易追蹤版本差異
- 文件比較好寫

## 建議修改順序

建議分三個階段處理。

### 階段一: 先修會影響正確性與穩定性的問題

- 修正 `market_alert_daemon.py` 的 state / 通知提交順序
- 修正 legacy direction mapping
- 補 `hourly_wait_reporter.py` 與 `market_alert_daemon.py` 的 loop-level 錯誤隔離

### 階段二: 補工程化基礎

- 新增 `.gitignore`
- 建立依賴檔
- 修正 `advisor_api_server.py` 的 `innerHTML`
- 補最小測試骨架與關鍵規則測試

### 階段三: 補結構化與可維護性

- 逐步模組化
- 抽共用常數與共用函式
- 持續整理設定與策略常數

## 每次修改後建議執行的驗證

### 靜態驗證

```powershell
python -m py_compile advisor_api_server.py market_alert_daemon.py shadow_mode.py hourly_wait_reporter.py alert_performance_tracker.py historical_replay_backtest.py report_alignment_backtest.py backfill_dashboard.py alert_timeline_report.py
```

### 核心流程驗證

```powershell
python shadow_mode.py BTC --llama off
python market_alert_daemon.py --config watchlist.json --state-db alert_state.db --interval-seconds 60 --llama off --once
python hourly_wait_reporter.py --config watchlist.json --once --llama off
python alert_performance_tracker.py --db-path alert_state.db --loop-minutes 0
```

### API 驗證

```powershell
python advisor_api_server.py --host 127.0.0.1 --port 8765 --llama off --risk-profile conservative
```

檢查項目:

- `/health` 是否回傳正常
- `/api/analyze` 是否能回傳結果
- 頁面是否能正確顯示中文
- 外部資料若含 HTML，頁面是否只顯示文字

## 補充建議

如果你只想先做最有價值的一輪修改，建議先完成以下六件事：

1. 修正 `market_alert_daemon.py` 的 state 與通知提交順序
2. 修正 `alert_performance_tracker.py` 的事件方向映射
3. 幫 `hourly_wait_reporter.py` 加上循環級錯誤保護
4. 幫 `market_alert_daemon.py` 加上 cycle-level 錯誤保護
5. 新增 `.gitignore` 與依賴檔
6. 修正 `advisor_api_server.py` 的 `innerHTML`

完成這一輪後，專案的實際穩定度會比現在明顯提升，而且不需要重構整個系統。
