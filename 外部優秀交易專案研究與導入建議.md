# 外部優秀交易專案研究與導入建議
更新日期：2026-03-20

## 研究目的

這份文件的目的不是找一個專案直接取代 `check_price`，而是找出成熟專案的可借鏡做法，判斷哪些適合導入目前這套：

- 多時框分析
- Telegram 提醒
- 事件分層
- 回測與成效追蹤

目前 `check_price` 的強項是：

- 技術分析與提醒流程已經成形
- 已有事件分層：`approach -> touch -> effective -> retest -> second_*`
- 已有 live 成效追蹤與 Telegram 工作流

目前較弱的部分是：

- 執行層仍偏手動
- 風控與停機機制仍不夠制度化
- 回測仍以自家 replay 為主，參數搜尋能力不足
- 自動交易需要的模組化程度還不夠

## 案例一：Freqtrade

### 它怎麼運作

Freqtrade 是完整的 crypto 交易 bot，功能從資料下載、回測、超參數搜尋、實盤交易、Web UI 到 Telegram 控制都已打通。

它的核心做法有幾個很值得注意：

- 用策略類別定義進出場條件
- 有明確的 callback 機制處理動態出場與倉位調整
- 有獨立的保護機制，不把所有風控都揉進策略訊號本身
- 有 `lookahead-analysis` 這種工具直接檢查偷看未來資料

官方文件裡比較重要的 callback 包含：

- `custom_stoploss()`
- `custom_exit()`
- `adjust_trade_position()`
- `confirm_trade_entry()`
- `confirm_trade_exit()`

此外，它還有獨立 `Protections`：

- `StoplossGuard`
- `MaxDrawdown`
- `LowProfitPairs`
- `CooldownPeriod`

### 值得學的地方

對 `check_price` 最有價值的不是整套搬進來，而是兩個觀念：

1. 訊號層和保護層分開  
現在 `check_price` 很多風險仍然混在事件條件裡。Freqtrade 的做法比較成熟：策略負責找機會，保護層負責說「現在先不要做」。

2. 出場要有 callback，而不是只靠固定 TP/SL  
你現在已經開始做保本上移、分批止盈，但還比較像報告文字。Freqtrade 的做法提醒我們：這類規則應該變成明確的執行函式。

### 對這個專案適不適合

適合借鏡，不適合直接改成 Freqtrade 專案。

原因：

- 你的專案目前不是「策略 class + 交易引擎」架構
- 直接遷移成本高，而且會把現有 Telegram / 報告流程打散

最適合導入的部分：

- 獨立的 `protections` 層
- 交易停機規則
- 連敗保護
- pair cooldown
- lookahead / recursive 檢查思維

## 案例二：Jesse

### 它怎麼運作

Jesse 比較像「研究、回測、優化、實盤」一體化的 crypto strategy framework。

它強調幾件事：

- 多時框、多標的同步回測
- 避免 look-ahead bias
- 支援 futures、槓桿、short
- 風控與 partial fills 是一級公民
- 有 paper trading 與 live trading

對你來說，Jesse 最重要的訊號不是它有哪些指標，而是它把「研究模式」和「執行模式」接得很近。

### 值得學的地方

1. 策略物件化  
目前 `check_price` 的核心邏輯還是偏報告函式流。Jesse 這種做法比較適合日後把訊號正式收斂成可執行策略。

2. 多時框回測做得更自然  
你現在本來就有 5m、1h、4h、多層事件，Jesse 這套思路與你很接近。

3. partial fills / 分批進出場  
你現在很重視「試單、確認、加碼」，這和 Jesse 的分批進出場哲學是同一路線。

### 對這個專案適不適合

中度適合，但適合作為中期重構方向，不是下一步就搬。

原因：

- 你現在已經有自己的分析與提醒邏輯
- 但還沒有整理成明確的 strategy object / trade lifecycle

如果未來要往半自動交易走，Jesse 的思想很值得學：

- 把事件變成策略狀態
- 把進場、加碼、出場寫成明確交易生命週期

## 案例三：Hummingbot

### 它怎麼運作

Hummingbot V2 的設計很值得注意，因為它把策略拆成三層：

- `Executors`
  - 負責一段有限任務，例如進場、DCA、TWAP、Grid
- `Scripts`
  - 簡單策略的入口
- `Controllers`
  - 長時間運行、可重用、可測試的策略協調層

官方文件明確寫到：

- Controller 透過 `MarketDataProvider` 取資料
- 再發出 `ExecutorActions`
- 讓上層去建立或停止 executor

這種設計非常像你現在專案最需要的下一步。

### 值得學的地方

1. 事件判斷和執行流程分層  
你現在已經有：

- `approach`
- `touch`
- `effective`
- `retest`
- `second_*`

這些其實已經很接近「Controller 產生信號，Executor 執行任務」的世界觀。

2. Executor 是有限生命週期  
這點非常重要。  
目前 `check_price` 的提醒會告訴你市場狀態，但還沒有把每一輪 setup 正式當成一個有限生命周期的對象。

3. 多策略並行  
如果未來你想同時跑：

- breakout
- retest
- mean reversion watch

Hummingbot 的控制器概念比你現在的單一函式流更合適。

### 對這個專案適不適合

非常適合借鏡架構，不適合整套替換。

我認為 Hummingbot 是和 `check_price` 最相近的參考案例，因為：

- 你目前也是事件驅動
- 也有提醒與長駐 daemon
- 也有「先觀察、再確認、再行動」的節奏

最值得導入的不是 market making 策略，而是這種拆法：

- `SignalController`
- `TradeExecutor`
- `RiskGuard`

## 案例四：QuantConnect LEAN

### 它怎麼運作

LEAN 是更底層、更工程化的量化交易引擎。

它把系統拆成明確插件：

- `IResultHandler`
- `IDataFeed`
- `ITransactionHandler`
- `IRealtimeHandler`
- `ISetupHandler`

而且它的 Algorithm Framework 還把交易邏輯拆成：

- Alpha
- Portfolio Construction
- Risk Management
- Execution

### 值得學的地方

1. 模組責任清楚  
目前 `check_price` 雖然已經比之前好很多，但 `shadow_mode.py` 仍然有點大。LEAN 這種責任分離很值得學。

2. 風控與執行獨立  
它不是讓 alpha 訊號直接等於下單，而是中間還有 portfolio / risk / execution。

3. 結果處理是獨立模組  
這對你很有啟發，因為你現在同時有：

- Telegram
- Markdown 報告
- DB 紀錄
- API

這些其實可以更像 `ResultHandler` 家族。

### 對這個專案適不適合

適合當長期架構參考，不適合短期直接引入。

原因：

- LEAN 太完整，也太重
- 你的專案目前還在「策略與提醒優化」階段，不值得直接升級成大型引擎

## 案例五：vectorbt / backtesting.py

### 它們怎麼運作

這兩個都偏研究工具，但定位不同：

- `vectorbt`
  - 向量化、快速、適合大量參數搜尋
- `backtesting.py`
  - 比較輕量、API 簡單、適合快速驗證單一策略

### 值得學的地方

1. 研究與驗證速度  
你現在已經有自己的 replay，但如果想大量測：

- `approach_proximity_pct`
- `retest_volume_ratio_min`
- trigger 距離
- stop / TP 比例

vectorbt 這種工具很適合。

2. 可快速做訊號 A/B  
如果你想驗證：

- `retest_hold` 要不要加 volume
- `approach_up` 是否應該再降權
- `second_breakdown_short` 在不同 regime 下是否穩

這種參數掃描，自己手寫 replay 會慢很多。

### 對這個專案適不適合

非常適合立即導入，尤其是 `vectorbt`。

因為你現在最大的需求之一就是：

- 更快比較版本
- 更快看哪個條件組合較穩

這比直接接真實下單更有價值。

## 綜合判斷

如果只問「哪個專案最值得你現在學」，我的排序是：

1. Hummingbot  
原因：架構思路最像你現在的事件驅動提醒系統

2. Freqtrade  
原因：保護機制、風控停機、callback 式出場非常實用

3. vectorbt  
原因：最適合立刻提升你的研究與回測速度

4. Jesse  
原因：適合中期把事件整理成正式交易策略物件

5. LEAN  
原因：適合長期架構參考，但現在太重

## 對 check_price 的具體建議

### 最適合現在立刻做的

#### 1. 加一層獨立風控保護模組

借鏡 Freqtrade。

建議新增：

- `StoplossGuard`
- `MaxDrawdownGuard`
- `PairCooldown`
- `HighVolatilityPause`

用途不是改技術分析，而是決定：

- 這段時間先不要再做
- 這個幣先冷卻
- 連續失敗後先停機

#### 2. 把現有事件正式拆成 Controller / Executor 思維

借鏡 Hummingbot。

建議未來拆成：

- `SignalController`
  - 負責產生 `approach / touch / effective / retest / second_*`
- `TradePlanBuilder`
  - 負責把事件轉成止損、目標、回踩區、破位後路徑
- `ExecutionExecutor`
  - 先做 paper / draft，不要直接真下單

這會讓你現在的 Telegram 提醒更容易升級成半自動掛單系統。

#### 3. 加一個參數研究層

借鏡 vectorbt。

建議優先拿來測：

- `approach_proximity_pct`
- `lock_timeout_minutes`
- `lock_drift_pct`
- `retest_volume_ratio_min`
- `effective` 量比門檻

這會比現在每次改完再手動 replay 更有效率。

### 中期值得做的

#### 4. 把交易生命週期寫清楚

借鏡 Jesse。

你現在已經有雛形，但還可以更正式：

- `watch`
- `armed`
- `probe_entry`
- `confirmed_entry`
- `add_on`
- `breakeven`
- `scaled_out`
- `closed`

現在很多文字描述其實都對應到這些狀態，只是還沒完全物件化。

#### 5. 把輸出層模組化

借鏡 LEAN 的 `ResultHandler` 思想。

建議拆成：

- Telegram formatter
- Markdown formatter
- API serializer
- DB writer

這樣之後你改報告格式時，不會一直去碰核心分析邏輯。

## 不建議現在做的事

### 1. 不要整套換成別人的框架

現在最不值得做的是：

- 全面遷移到 Freqtrade
- 全面遷移到 Jesse
- 直接導入 LEAN 當主引擎

原因很簡單：

- 你的專案已經有自己的優勢
- 最近幾輪也已經把事件流與回測邏輯修順了
- 現在該做的是局部借鏡，不是推翻重建

### 2. 不要太早接真實自動下單

在借鏡成熟專案之前，最容易犯的錯就是覺得：

- 既然提醒越來越像樣
- 那就快點接交易所 API

但目前更值得先做的是：

- protections
- paper executor
- parameter lab

## 我對這個專案的下一步建議

如果只選三件最值得做的，我會選：

1. 導入 Freqtrade 式 `protections` 概念  
讓風控與停機不再散落在訊號邏輯裡

2. 依 Hummingbot 思路，把事件和執行拆成 `controller + executor`  
讓這套系統能從提醒工具走向半自動系統

3. 用 vectorbt 建一個快速參數研究腳本  
專門測目前幾個最敏感參數

## 結論

`check_price` 現在不需要被別的框架取代，反而已經有自己的特色：

- 報告導向
- Telegram 導向
- 事件分層明確
- 支撐壓力地圖與破位路徑已經很實戰

真正該做的是：

- 向成熟框架借架構
- 向成熟框架借風控
- 向成熟研究工具借驗證速度

我現在的主張很明確：

- 短期：學 `Freqtrade + vectorbt`
- 中期：學 `Hummingbot + Jesse`
- 長期：參考 `LEAN`

這樣最務實，也最不會把你已經做出來的優勢打掉重練。
