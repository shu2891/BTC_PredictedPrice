# 市場樣本庫與條件機率分析方案

## 背景

目前 `check_price` 已經有：

- 即時事件提醒
- 事件分層
- live 成效回填
- 歷史 replay 回測

但現在的分析仍有一個結構性限制：

- 我們知道某類事件最近有沒有用
- 但還不夠知道「它在什麼市場條件下更容易成功或失敗」

例如：

- `effective_long_breakout` 在真上破日可能有效
- 同一個事件在 `range_or_mixed + high risk` 盤裡可能失敗

這代表目前專案還缺一層：

> 專門記錄當下市場特徵，並用條件機率去解讀事件 edge 的樣本庫與分析層

## 目標

新增一套「市場樣本庫 + 條件機率分析」能力，讓專案從：

- 事件提醒系統

進一步升級成：

- 可以回答「在某種市場條件下，某類事件 historically 成功率與期望值如何」的決策系統

核心不是讓 AI 更會猜，而是讓系統更：

- 可驗證
- 可比較
- 可校準
- 可持續變準

## 想解決的具體問題

### 1. 目前事件有成效，但缺少完整市場上下文

現在資料庫重點在：

- 事件類型
- 送出時間
- 後續 horizon 表現

但還缺很多會影響結果的條件，例如：

- 當下 `market_regime`
- 是否高波動
- 與上下破價的距離
- 支撐 / 壓力結構
- `executor_plan.quality`
- 是否有 protection layer
- 是否處於消息面高風險時間窗

### 2. 現在的統計仍偏「平均表現」

目前我們常看：

- 哪類事件平均報酬較高
- 哪類事件近幾天比較準

但實戰真正更重要的是：

- `P(1h > 0 | event=effective_long_breakout, regime=range_or_mixed, risk=high)`
- `P(stop_loss | event=retest_hold_long, symbol=SOLUSDT)`
- `P(tp1_hit | event=second_breakdown_short, protection=normal)`

也就是：

> 需要從平均值分析，升級成條件機率分析

### 3. 目前回顧仍偏人工解讀

現在每次回顧都要重新看：

- 當時市場是什麼結構
- 那個事件是在什麼情境下發生
- 它為什麼成功或失敗

這件事應該變成系統可查詢，而不是每次手動重新推理。

## 參考的外部優秀方案

這次方案主要參考 5 類成熟做法。

### 1. Freqtrade：把交易結果與信號導出，做後續分析

參考：
- [Freqtrade Backtesting](https://docs.freqtrade.io/en/latest/backtesting/)
- [Freqtrade Advanced Backtesting Analysis](https://docs.freqtrade.io/en/2024.6/advanced-backtesting/)

值得借鏡的地方：

- 回測結果不是只看總績效
- 會把 `signals / trades / tags` 導出給後續分析
- 方便依進場原因、出場原因、標的、條件做二次統計

對本專案的啟發：

- `check_price` 不應該只存「事件有沒有成功」
- 還應該存「事件當時的市場特徵」

### 2. vectorbt：把訊號與記錄視為一級分析對象

參考：
- [vectorbt Features](https://vectorbt.dev/getting-started/features/)
- [vectorbt Signals](https://vectorbt.dev/api/signals/)
- [vectorbt Records / Stats](https://vectorbt.dev/api/records/base/)
- [vectorbt Portfolio](https://vectorbt.dev/api/portfolio/base/)

值得借鏡的地方：

- `signals` 是一級公民
- `records` 可以直接做 stats / metrics / group analysis
- 可以針對事件型資料做快速聚合，而不只看最後績效

對本專案的啟發：

- `approach / touch / effective / retest / second_*` 應該被當成結構化 records
- 後續分析應該圍繞這些 records 做條件統計

### 3. DuckDB：適合本地分析型資料倉

參考：
- [DuckDB Parquet Overview](https://duckdb.org/docs/stable/data/parquet/overview)
- [DuckDB Timestamp Types](https://duckdb.org/docs/stable/sql/data_types/timestamp.html)

值得借鏡的地方：

- 適合本地分析 workloads
- 可直接查 Parquet
- 有 filter / projection pushdown
- 很適合把大量歷史樣本做離線統計

對本專案的啟發：

- `SQLite` 很適合現在的 operational event store
- 但如果要做大量歷史條件分析，`DuckDB + Parquet` 會更合適

### 4. Polars：適合做 lazy / batch 條件分析

參考：
- [Polars Lazy API](https://docs.pola.rs/user-guide/concepts/lazy-api/)
- [Polars Using the Lazy API](https://docs.pola.rs/user-guide/lazy/using/)
- [Polars Getting Started](https://docs.pola.rs/user-guide/getting-started/)

值得借鏡的地方：

- lazy query
- schema 檢查
- 對大樣本聚合與分桶分析很方便

對本專案的啟發：

- 條件機率分析層不應該直接寫成一堆 Python loop
- 可以考慮用 `Polars LazyFrame` 做規模化聚合與特徵分桶

### 5. Timescale：給未來擴張的 time-series 方向

參考：
- [Timescale Hypertables](https://docs.timescale.com/use-timescale/latest/hypertables/)

值得借鏡的地方：

- 真正大規模 time-series 的資料模型
- chunk / hypertable 思路適合長期累積市場快照

但對本專案目前的判斷：

- 現階段不需要直接導入
- 運維成本太高
- 對單機 Python 專案來說，`DuckDB + Parquet` 更務實

## 建議的整體架構

### 原則

不要推翻現在的 `SQLite operational store`。

應該改成雙層：

### Layer 1：Operational Store（保留現況）

用途：

- 即時提醒
- state machine
- live event persistence
- 短期回填

目前資料庫：

- `alert_state.db`
- `shadow_mode.db`

### Layer 2：Analytical Store（新增）

用途：

- 市場樣本庫
- 條件機率分析
- 歷史查詢
- 特徵分桶統計

建議技術：

- `DuckDB` 作為分析層資料庫
- `Parquet` 作為長期樣本檔案格式

原因：

- 不破壞現有 live 邏輯
- 分析可以快很多
- 後續要加欄位或重跑統計比較方便

## 建議新增的資料模型

### 1. `market_context_snapshots`

每次事件發生時，寫一筆「當時市場上下文」。

建議欄位：

- `snapshot_id`
- `event_id`
- `symbol`
- `snapshot_time`
- `event_type`
- `direction`
- `decision`
- `market_regime`
- `risk_level`
- `analysis_bias`
- `direction_bias`
- `execution_readiness`
- `returns_5m`
- `returns_1h`
- `volatility_24h`
- `tf5_trend`
- `tf1h_trend`
- `tf4h_trend`
- `volume_ratio_5m`
- `volume_ratio_1h`
- `above_vwap_5m`
- `breakout_up`
- `breakout_down`
- `distance_to_breakout_up_pct`
- `distance_to_breakout_down_pct`
- `distance_to_long_ready_zone_pct`
- `distance_to_short_ready_zone_pct`
- `price_phase`
- `primary_support`
- `secondary_support`
- `primary_resistance`
- `secondary_resistance`
- `protection_status`
- `protection_rules_json`
- `executor_quality`
- `executor_rr_tp1`
- `executor_cancel_after_minutes`

### 2. `event_outcomes`

把現有 `alert_event_performance` 抽成更分析友善的結構。

欄位可包含：

- `event_id`
- `horizon_label`
- `close_return_pct`
- `max_runup_pct`
- `max_drawdown_pct`
- `tp1_hit`
- `tp2_hit`
- `stop_loss_hit`
- `outcome_label`

### 3. `market_regime_snapshots`（可選）

如果未來希望做非事件型分析，可以固定時間寫市場快照。

建議頻率：

- 每 1 小時

用途：

- 回頭查當時市場是否在轉 regime
- 看橫盤持續多久
- 看高波動是否擴散到多個標的

這層不是第一優先，但長期很有價值。

## 建議新增的功能

### A. 歷史市場資料查詢

功能：

- 查某個事件
- 查某個標的
- 查某個日期區間
- 查某種市場狀態

查詢例子：

- `SOLUSDT` 在 `range_or_mixed` 裡的 `effective_long_breakout`
- `ETHUSDT` 在 `high risk` 時的 `retest_hold_short`
- `second_breakdown_short` 在 `protection=guarded` 時的表現

### B. 條件機率報告

輸出：

- `P(close_return_pct > 0 | conditions)`
- `P(tp1_hit | conditions)`
- `P(stop_loss_hit | conditions)`
- `avg close_return_pct | conditions`
- `avg max_runup_pct / max_drawdown_pct | conditions`

### C. 市場型態樣本回顧

用來回答：

- 真上破日長什麼樣
- 混合盤假突破日長什麼樣
- 空方延續盤長什麼樣

這比只看單日 summary 更有訓練價值。

## 建議的分析邏輯

### Phase 1：先用頻率派條件機率

先不要急著上機器學習。

第一版直接做：

- 分桶統計
- 條件機率
- 不同 regime 下的成功率 / 失敗率

這一層最穩，也最容易解釋。

### Phase 2：再做校準

當樣本夠多時，再做：

- reliability / calibration
- 不同事件 quality bucket 的校準

例如：

- `executor_quality = tradable` 是否真的比 `caution` 更好
- `effective_long_breakout` 的成功率是否被高估

### Phase 3：最後才考慮模型化

只有當資料量夠時，才值得考慮：

- logistic regression
- tree-based ranking
- Bayesian shrinkage

目前不建議一開始就走 ML，因為樣本量還不夠大。

## 我對技術選型的建議

### 現階段最佳做法

- 保留 `SQLite` 做即時 operational store
- 新增 `DuckDB + Parquet` 做 analytical store
- 用 `Polars` 做離線聚合與條件統計

### 不建議現在直接做的事

- 不建議直接改成 Timescale / PostgreSQL
- 不建議現在就做複雜 ML 模型
- 不建議先做漂亮 UI，再補資料模型

## 專案內最值得優先補的地方

### 1. 事件發生時的市場上下文快照

這是整個方案最重要的一步。

如果不先把當時市場特徵存下來，之後再多的分析都只是事後猜測。

### 2. 條件機率報告腳本

例如新增：

- `market_context_archive.py`
- `conditional_probability_report.py`

### 3. 歷史查詢輸出

先做：

- `md`
- `json`

先不做 UI。

## 分階段執行方案

### 第一階段：資料先存對

新增：

- `market_context_archive.db` 或 `market_context.duckdb`
- 每個 event 對應一筆 snapshot

交付：

- 樣本表
- schema
- 單次匯出工具

### 第二階段：條件機率分析

新增：

- `conditional_probability_report.py`

輸出：

- 依事件 / 幣種 / regime / protection / executor quality 的條件統計

### 第三階段：歷史回顧查詢

新增：

- 給人看的 `md`
- 給程式用的 `json`

### 第四階段：再決定要不要做 UI 或模型化

## 對專案的直接價值

如果這個方案做出來，專案會得到三個很實際的提升：

### 1. 對市場的看法會更「有根據」

不是只說：

- 這幾天覺得 long 好像比較弱

而是能說：

- 在 `range_or_mixed + high risk + alt` 的條件下，`effective_long_breakout` 過去 `N` 筆中 `1h` 正報酬率只有多少

### 2. 更容易知道該刪什麼

某些事件若長期在某些條件下失效，就能明確降級或停用。

### 3. 更容易知道該維護什麼

某些事件如果只在特定市場型態有 edge，就能更有信心保留。

## 我的最終建議

這個方案值得做，而且是目前最正確的下一步之一。

但要記住：

- 先做資料模型
- 再做條件機率
- 最後才做更複雜的模型或 UI

如果直接跳到機器學習或花俏 dashboard，很容易把事情做反。

## 我會怎麼開始

如果現在就開始動手，我會按這個順序做：

1. 新增市場上下文快照 schema
2. 在 live event 送出時把 snapshot 寫入分析層
3. 寫第一版 `conditional_probability_report.py`
4. 用目前已累積的 live event 先出第一份條件機率報告

這樣可以最快驗證這個方向是否真的幫助專案變準。
