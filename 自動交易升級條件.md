# 自動交易升級條件

更新日期: 2026-03-18

## 目的

這份文件整理 `check_price` 從「分析與提醒工具」升級到「可自動交易系統」前，還缺哪些條件。

重點不是先接交易所 API，而是先確認：

- 訊號是否夠穩
- 進出場是否可執行
- 風控是否完整
- 系統是否能承受異常

目前的判斷是：

- 適合做 `半自動決策輔助`
- 暫時 `不建議直接做全自動真倉交易`

## 目前狀態

目前專案已具備：

- 多標的即時分析
- 上下破價與短線執行區
- Telegram 主動提醒
- 事件狀態追蹤
- 報告輸出
- 初步的方向與可交易性分離

目前專案仍缺：

- 穩定的自動下單規則
- 完整的執行風控
- 模擬成交驗證
- 系統級異常保護
- 足夠長期的策略驗證

## A 級條件

這些沒做到前，不建議直接自動下單。

### 1. 進場條件固定化

不能只因為價格碰到上破價或下破價就進場。

至少要明確定義：

- 首次突破 / 首次跌破
- 回踩守住 / 反抽失敗
- 第二次突破 / 第二次跌破
- 失敗位

如果沒有把這四個條件明確寫成規則，機器很容易在假突破時直接追單。

### 2. 風險控制固定化

自動交易不能只看方向，要先限制最大可承受損失。

至少要固定：

- 單筆最大帳戶風險
- 每日最大虧損
- 最大同時持倉數
- 同標的重複進場限制
- 連敗停機條件

沒有這一層，策略就算方向邏輯正確，也可能因為單次劇烈波動造成大回撤。

### 3. 下單執行防呆

策略判斷和交易所執行是兩回事。

自動下單前要先確認：

- 最小下單量
- 價格精度與數量精度
- 逐倉 / 全倉模式
- 槓桿設定
- 是否已有同方向倉位
- 是否允許反手

如果沒有這些檢查，實際下單時會遇到拒單、重複持倉或錯誤倉位問題。

### 4. 狀態一致性

系統要能保證：

- 訊號送出
- 下單成功
- 成交完成
- 止損 / 止盈掛出
- 狀態寫入資料庫

這些狀態彼此一致。

否則會出現：

- Telegram 有提醒，但系統沒有記錄
- 系統以為已進場，但交易所其實沒成交
- 系統以為已出場，但倉位仍在

### 5. 模擬成交層

在真倉前，必須先有 paper trading 或模擬成交。

至少要能模擬：

- 手續費
- 滑價
- stop-market / stop-limit 差異
- 掛單未成交
- 部分成交

否則回測與實盤會差太多。

## B 級條件

這些做完後，策略才有機會從「可用」變成「較穩定」。

### 6. 方向與可交易性分離

目前已經開始做這件事，但還要持續校準。

核心觀念是：

- `偏多` 不等於 `現在可以追多`
- `偏空` 不等於 `現在可以直接空`

要讓報告能區分：

- 分析方向
- 當下是否值得執行

### 7. 出場邏輯完整化

看對方向不代表能賺到錢。

至少要補強：

- 保本上移條件
- 分批止盈條件
- 假突破撤退條件
- 失敗後不立刻反手的條件

這一層決定的是「拿得住」而不是「看得準」。

### 8. 市場狀態分層驗證

不能只看最近一段多頭行情。

至少要拆成：

- 多頭
- 空頭
- 盤整

分開驗證策略表現。

不然現在看起來準的條件，到了另一種市場環境可能完全失效。

### 9. 真實成本回測

回測不能只看理想價格命中。

要納入：

- 交易手續費
- 點差
- 滑價
- 追價成交品質
- 觸發後未成交

這些會直接改變策略最終是否有正期望值。

### 10. 異常處理

長時間運行的策略一定會遇到外部失敗。

要先處理：

- API timeout
- RSS 抓取失敗
- Telegram 發送失敗
- 交易所拒單
- 資料缺失
- 程式重啟恢復

沒有這層，系統就算邏輯正確，也不適合自動化。

## C 級條件

這些是工程成熟度，適合在核心正確性穩住後補上。

### 11. 自動化測試

優先測這些核心規則：

- alert state 狀態轉移
- event direction 映射
- 風控條件
- 出場條件
- 假突破過濾
- Telegram 失敗時的狀態一致性

### 12. 版本追蹤

每筆策略判斷與每筆交易，都要知道：

- 使用哪個版本的策略
- 使用哪組參數
- 當時的市場資料

這樣才能回頭檢討，而不是只看結果。

### 13. 監控與告警

系統要能主動告訴你：

- 策略停止運行
- 連續虧損過多
- 訂單異常
- 長時間沒有成交
- 外部依賴故障

### 14. 參數集中管理

目前很多數值仍散在程式裡。

建議集中管理：

- proximity
- ATR 倍數
- second breakout 條件
- retest failure 條件
- volume ratio 門檻
- lock timeout / drift

這樣才方便做版本比較與調參。

## 建議升級路線

不要直接從提醒系統跳到全自動真倉。

建議分四步走：

### 第一步：半自動

機器負責：

- 分析
- 提醒
- 列出掛單草稿

人負責：

- 最後確認是否進場

### 第二步：模擬下單

機器開始根據規則記錄：

- 會在哪裡進
- 會在哪裡出
- 會承受多少風險

但不連交易所。

### 第三步：小資金真倉

先用很小的資金驗證：

- 成交品質
- 執行延遲
- 下單與止損是否穩定

### 第四步：全自動

只有在前三步都穩定後，才考慮：

- 自動進場
- 自動止損
- 自動止盈
- 自動風控停機

## 目前最值得先補的四件事

如果只先做一輪最有價值的升級，建議順序是：

1. 修正 `market_alert_daemon.py` 的 state 與通知提交順序
2. 補 `hourly_wait_reporter.py` 和 daemon 的循環級錯誤隔離
3. 建立模擬成交與風控層
4. 把回踩 / 二突後才允許下單的規則正式寫進執行邏輯

## 結論

這個專案現在最適合的定位是：

- `半自動交易輔助系統`

還不是：

- `可放心全自動真倉交易系統`

如果後續要持續升級，方向不是再加更多指標，而是先補：

- 執行規則
- 風險控制
- 狀態一致性
- 模擬成交

把這四件事做紮實之後，自動交易才有意義。
